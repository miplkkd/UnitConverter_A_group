## [User / Role 2] Turn 1 답변

> 과제 PDF 링크 받고 README부터 읽었어요. 출력 예시(meter:2.5 → feet, yard…)랑 OCP, SRP, 테스트 쓰라는 말이 먼저 눈에 들어왔고, 추가 요구(JSON, 동적 단위 등록)는 “시간 남으면” 쪽이라 일단 무시하고 기본만 하기로 했어요.
>
> 그다음 바로 UnitConverter.py 열었어요. main()에 if-elif로 meter/feet/yard 나뉜 걸 보고 “이걸 그대로 두면 OCP 안 된다”는 느낌만 들었지, 어떻게 쪼갤지는 아직 몰랐어요.
>
> 터미널에서 한 번 실행해 봤어요. meter:2.5 넣으면 세 줄 나오는 건 확인했고, abc:1이나 meter:hello 넣었을 때 어떻게 깨지는지도 손으로 먼저 봤어요. 그때 “입력 검증”이 README에 있는 이유를 체감했어요.
>
> 테스트부터 쓰진 않았어요. pytest 폴더 구조가 헷갈려서, 그날은 변환 숫자 맞는지 계산기로 2.5m → feet 대략 맞는지만 확인했어요.
>
> AI는 그 시점엔 안 썼어요. “인터페이스 어떻게 나누지”만 검색해 보다가, 예제가 너무 길어서 과제 마감 전날에야 “meter 기준으로 변환 클래스 예시” 정도 물어봤어요.
>
> 정리하면 순서는: README → 코드 열기 → 실행·잘못된 입력 실험 → 손계산 확인 → (테스트·리팩터링은 그다음 날) 이었어요.

> 당시 판단:

> - “지금 막히는 건 if-elif 리팩터링”
> - “README 숫자랑 출력만 맞으면 발표 때 덜 당황”
>
> conftest, coverage, CI는 그날 전혀 안 건드렸어요.
